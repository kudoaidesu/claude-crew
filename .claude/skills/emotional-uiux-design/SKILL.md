# 感情駆動型 UI/UX デザイン

## なぜこのフローが最高の結果を生むのか

### 核心的な成功要因

**1. 「中から外へ」の設計順序**
普通のUI/UXフローは「画面→機能→動線」と外側から攻める。このフローは**「ビジョン→視覚基盤→感情→数値→表示→導線→実装」**と、ユーザーの内面から外側へ設計する。結果、すべてのUIパーツに「なぜこれが必要か」の根拠がある。根拠のないUIが存在しない。

**2. 5部門トップ × ペルソナの交互レビュー**
- **部門トップだけ**だとプロの目線で美しくなるが、ユーザーが使えない
- **ペルソナだけ**だと要望の寄せ集めで一貫性がなくなる
- **交互にレビュー**すると「プロの構造 × ユーザーの直感」の両方が満たされる

**3. ペルソナの多様性**
6つの異なる検証軸でテストする。それぞれが異なる観点を検証する：
- **最短パスの検証** — 面倒な操作を避けるユーザー
- **情報密度の検証** — 時間がないユーザー
- **感情的受容性の検証** — 表現に敏感なユーザー
- **美的品質の検証** — デザインの洗練を重視するユーザー
- **上級効率の検証** — 効率とカスタマイズを求めるユーザー
- **アクセシビリティの検証** — 支援技術を使うユーザー

**4. リフレーミングの力**
「低い評価」→「伸びしろ」のような言い換えだけで、ユーザーの反応が「ムカつく→直したい」に変わる。UIの形ではなくコピーが感情を決める。

## 完全ワークフロー（9ステップ）

### Phase 1: 構造化（What）

#### Step 1: 機能・画面整理
```
入力: プロダクトのビジョン文書、既存仕様
出力: 機能一覧 × 必要画面の対応表
```
- プロダクトのビジョン・コアバリューを確認する
- 全機能を洗い出し、MVPに必要な範囲を決める
- 各機能に必要な画面を特定する
- タブ構成・画面遷移の骨格を設計する

#### Step 2: キーメトリクス洗い出し
```
入力: 機能一覧
出力: 運営KPI × ユーザー体験メトリクスの2軸マトリクス
```
- 各機能について運営側のKPI（コンバージョン率、離脱率等）を定義する
- ユーザーが**画面上で見る数値**を別途洗い出す
- ビジョン達成に必要なユーザー体験メトリクスを論理的に導出する

#### Step 3: 視覚基盤の方向性導出
```
入力: プロダクトのビジョン・コアバリュー（Step 1で確認済み）
出力: 5領域の視覚方向性
```
プロダクトのビジョンを3-5の形容詞で表現し、以下の方向性をトップダウンに導出する：
1. **タイポグラフィの性格** — 書体カテゴリと密度の方向
2. **スペーシング密度** — コアタスクの頻度・深さに応じた密度レベル
3. **深さ表現** — シャドウ/ボーダーの強度と手法
4. **レスポンシブ起点** — コアタスクが発生するデバイスで起点を決定
5. **ダークモード方針** — 利用シーンから対応要否と時期を判断

> [!NOTE]
> ここでは具体値（フォント名、px値等）は決めない。「方向性」のみ。
> 具体値はStep 7の5部門レビューでUI部門長・デザインシステム部門長が決定する。

### Phase 2: 感情設計（Why）

#### Step 4: 感情データポイント洗い出し
```
入力: ユーザー体験メトリクス
出力: 感情カテゴリ別データポイント一覧
```
- ユーザーが見て**心が動く数値**を洗い出す
- 7つの感情カテゴリに分類する:

| カテゴリ | 意味 | 行動トリガー例 |
|---------|------|--------------|
| 衝撃 | 認識のギャップ | 「こんなに差があったのか」→ 現状を変えたい |
| 希望 | 改善の可能性 | 「ここを直せば良くなる」→ 行動を起こす |
| 嫉妬 | 社会的比較 | 「あの人はできている」→ 自分もやりたい |
| 怒り | 不満・乖離 | 「おかしいだろ」→ 改善を要求する |
| 感動 | 成長実感 | 「こんなに成長した」→ 継続する |
| 誇り | 自己肯定 | 「自分はすごい」→ シェアする |
| 焦り | 時限性 | 「今やらないと間に合わない」→ すぐ行動する |

- 各データポイントの行動トリガーを定義する

> [!IMPORTANT]
> ネガティブ感情（怒り・衝撃）のデータは必ずポジティブ導線（改善CTA）とセットにする。
> 単独で見せると離脱リスク。

#### Step 5: 表示方法のWEB調査
```
入力: 感情データポイント一覧
出力: 各データの最適表示パターン（調査根拠付き）
```
- 1つ1つのデータポイントについてWEB検索で最適な表示方法を調査する
- 心理学的根拠を添える
- 調査すべきキーワード例:
  - `animated counter emotional impact data visualization`
  - `gauge chart vs number display effectiveness`
  - `radar chart mobile app best practice`
  - `before after comparison graph user engagement`
  - `countdown timer FOMO urgency UX`
  - `social feed benign envy motivation design`
  - `celebration animation confetti UX milestone`

#### Step 6: ユーザー導線設計
```
入力: 表示方法一覧
出力: 4層設計（表示方法 × 心理根拠 × 導線 × 次CTA）
```
- 各データポイントを4層で整理する:
  1. 表示方法（UIパターン）
  2. なぜ効くか（心理学的根拠）
  3. 導線設計（どうやって自然にたどり着かせるか）
  4. 次のアクションCTA（見た後にどこへ誘導するか）
- 導線調査キーワード例:
  - `nudge theory mobile app behavior design`
  - `progressive disclosure onboarding`
  - `push notification optimal timing habit loop`
  - `CTA placement thumb zone mobile`
  - `emotional sandwich negative feedback UX`

### Phase 3: 反復レビュー（Validate）

#### Step 7: 5部門レビュー → ペルソナテスト → 繰り返し

**サイクル構造:**
```
5部門レビュー → 改善 → ペルソナテスト → 改善 → 5部門再レビュー → ...
                     パスするまで繰り返す
```

**5部門レビュアー:**

| レビュアー | 観点 | よくある指摘 |
|-----------|------|-------------|
| UI部門長 | 視覚的一貫性、カラー体系、アイコン体系、タイポグラフィ体系、情報階層 | カラー体系の不統一、アイコンスタイルの混在、フォント選定の不適切、チャート選択、ブランド整合性 |
| デザインシステム部門長 | デザイントークン、スペーシング体系、エレベーション体系、ダークモード | トークン未定義、スペーシングの不統一、シャドウレベルの過不足、ダークモード対応漏れ |
| モバイルデザイン部門長 | パフォーマンス、タッチ操作、プラットフォーム準拠、レスポンシブ戦略 | バッテリー消費、タッチターゲットサイズ、ボトムシートvsフルスクリーン、ブレークポイント設計 |
| UX部門長 | 行動心理、離脱リスク、導線設計、感情バッファ | 感情バッファの欠如、通知過多、空状態未設計、導線の断絶 |
| アクセシビリティ部門長 | 3重エンコード、支援技術対応、認知負荷、運動障害対応 | 色だけの情報伝達、スクリーンリーダー未対応、タッチターゲット不足、認知負荷過多 |

**UI部門長の決定事項:**

初回レビューでStep 3の方向性を具体値に落とし込み、プロダクト全体で統一する：

A. **アイコン体系**
1. スタイル選定 — ラインアイコン / フィルドアイコン / デュオトーン等
2. ライブラリ選定 — Lucide, Heroicons, Phosphor Icons等から1つ選ぶ
3. 線幅・サイズの統一ルール — デフォルト線幅とサイズバリエーション（sm/md/lg）を定義
4. セマンティックカラーとの連動 — アイコン色はカラー体系に従う

B. **タイポグラフィ体系**
1. フォントファミリー選定 — Step 3の方向性に基づき、見出し用・本文用を選定
2. タイプスケール定義 — サイズの段階（xs/sm/md/lg/xl/2xl等）
3. ウェイト・行間ルール — 各用途のウェイトと行間の基準

**デザインシステム部門長の決定事項:**

C. **スペーシング体系**
1. 基本単位の決定 — 4pt/8ptベースの選択
2. スケール定義 — 使用するスペーシング値の一覧

D. **エレベーション体系**
1. レベル数の決定 — 通常4-6段階
2. シャドウ/ボーダーのトークン定義

**ペルソナテスト:**

プロダクトに合わせてペルソナを設定し、以下の6つの検証軸をカバーする：

| 検証軸 | ペルソナ特性 | 典型的なフィードバック |
|--------|------------|---------------------|
| 最短パス | 面倒な操作を避ける | 「全部見る気がない」「一番大事なのだけ見せて」 |
| 情報密度 | 忙しく時間がない | 「何度もタップさせないで」「通知だけで完結して」 |
| 感情的受容性 | 表現に敏感 | 「要は私どうすればいいの？」「ネガティブな表現を使うな」 |
| 美的品質 | デザインの洗練を重視 | 「なんか安っぽい」「フォントが合ってない」 |
| 上級効率 | ショートカット・カスタマイズを求める | 「カスタマイズさせて」「キーボードで操作したい」 |
| アクセシビリティ | 支援技術を使用 | 「スクリーンリーダーで読めない」「コントラストが低い」 |

**レビュー時の調査キーワード:**
- `cognitive load reduction mobile app busy professionals`
- `lazy user design pattern one tap action reduce friction`
- `empty state first time user zero data design`
- `bottom sheet vs full screen modal iOS Android`
- `accessibility color blind data visualization triple encoding`
- `design token architecture primitive semantic component`
- `wcag 2.2 aria patterns screen reader testing`
- `aesthetic usability effect perceived quality UX`

**パス基準:**
- 5部門トップが全員パス AND 全ペルソナが「使いたい」と言うまで繰り返す

### Phase 4: 実装（Build）

#### Step 8: デザイン仕様の最終化
```
入力: 全レビュー通過したデザイン
出力: 実装可能な仕様書（セマンティックカラー、コンポーネント仕様、画面構成図）
```

#### Step 9: プロトタイプ実装 → ブラウザ検証
```
入力: 最終仕様書
出力: 動作するHTML/CSS/JSプロトタイプ
```
- CSS → HTML → JS の順に実装する
- ブラウザで全画面・全インタラクションを検証する
- 実装品質は **[frontend-design](../frontend-design/SKILL.md)** スキルを参照
- 仕上げは **[design-polish](../design-polish/SKILL.md)** スキルを参照

## 設計原則

### 5つの導線原則

| # | 原則 | 説明 |
|---|------|------|
| 1 | **2秒で要点** | AIサマリー1行で「今何すべきか」がわかる |
| 2 | **1タップで次へ** | カード全体タップ → 詳細展開 |
| 3 | **アプリ不要の情報伝達** | リッチ通知・ウィジェットで数字プレビュー |
| 4 | **ネガティブをリフレーム** | 「低い」→「伸びしろ」、「失敗」→「改善余地」 |
| 5 | **感情サンドイッチ** | 強み → 伸びしろ → 改善CTA |

### セマンティックカラー体系（テンプレート）

プロダクトに合わせてカスタマイズすること：

| 意味 | 用途例 |
|------|--------|
| 成長・ポジティブ | 評価上昇、目標達成、成功 |
| 注意・中立 | 乖離、期限接近、要確認 |
| 危険・ネガティブ | 低評価、緊急、エラー |
| 情報・補助 | ラベル、メタ情報、注釈 |
| 特別・祝福 | 限定機能、マイルストーン達成 |

### アクセシビリティ原則
- 色だけで情報を伝えない → **色 + アイコン + テキストラベル**の3重エンコード

## 使い方

**新しい画面・機能を追加する場合:**
1. **Step 1-2** — 機能とメトリクスを整理する
2. **Step 3** — ビジョンから視覚基盤の方向性を導出する
3. **Step 4** — その機能がどんな感情を生むか考える
4. **Step 5** — 最適な表示方法をWEB調査する
5. **Step 6** — 導線を設計する
6. **Step 7** — 5部門レビュー × ペルソナテストを回す
7. **Step 8-9** — 実装・検証する

**既存画面を改善する場合:**
1. 6ペルソナでテストする
2. フィードバックを分類する（最短パス？情報密度？感情？美的品質？上級効率？アクセシビリティ？）
3. 改善して5部門レビューに出す
4. パスするまで繰り返す
