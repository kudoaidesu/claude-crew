# デザインの仕上げ原則

## UI作成後は必ずこれを実行する

### 全フロー巡回スクリーンショットレビュー

プロトタイプ・実装が完了したら、**静的レビューではなく実際に操作して全状態を記録する**。

```
1. ブラウザでURLを開き、初期画面でスクリーンショット
2. すべてのタブ・サブ画面に移動 → 各画面でスクリーンショット
3. すべてのインタラクションフロー（ボタン押下・フォーム送信・アニメーション完了後）を
   1ステップずつたどり → 各ステップでスクリーンショット
4. スクロール有無を確認（コンテンツが隠れていないか）
5. 撮ったスクリーンショット全体を見て問題を**一覧化**
6. checklist.md と照合
7. まとめて修正を適用
```

**原則**: 「動作確認済み」≠「視覚的に問題なし」。静的コードレビューでは発見できない問題（テキスト折り返し、要素の重複、スクロール不足）はスクリーンショットでしか気づけない。

**やらないとどうなるか**: FABがカードに重なっている、長い名前が2行になりレイアウトが崩れている、祝福絵文字が小さい、などを実装後まで気づけない。

## 核心

> 「動く」と「仕上がっている」は違う

プロの納品物には、すべての要素に**意図**がある。

## 5つの仕上げ原則

### 1. すべての要素に居場所を与える

情報をグループ化し、階層を与える。

```
タイトル（大）> 日時・場所（中）> 補足情報（小・強調色）
```

### 2. セクションに役割を与える

背景色・ボーダー・ヘッダーでゾーニングする。

```tsx
<aside className="bg-white rounded-2xl border shadow-sm">
  <div className="bg-gradient-to-r from-blue-50 to-indigo-50 border-b">
    <Icon /> ラベル
  </div>
  <Content />
  <div className="border-t bg-gray-50">{/* 凡例 */}</div>
</aside>
```

### 3. 余白を意図的に設計する

- 関連要素は近く（4-8px）
- 異なるグループは離す（16-24px）
- セクション間はさらに離す（32-48px）

### 4. インタラクションで応答性を示す

```tsx
<Link className={cn(
  "group transition-all duration-200",
  "hover:shadow-lg hover:-translate-y-0.5",
  "focus-visible:ring-2",
  "active:scale-[0.99]"
)}>
  <img className="group-hover:scale-105" />
  <h3 className="group-hover:text-blue-600" />
</Link>
```

### 5. アクセシビリティは品質の証明

- タッチターゲット: `min-h-[44px]`
- ナビゲーション状態: `aria-current="page"`
- 画像CLS防止: `width={128} height={128}`

## ⚠️ UI修正時の原則

> **UI決定を最優先、コード品質系は後回し**

UI修正中に以下を同時にやらない：
- 未使用import削除
- タイムゾーン修正
- 型修正
- リファクタリング

**理由**: UIが変わるとコード自体が不要になる可能性がある。

**正しい順序**:
1. レイアウト・構造・余白を決定
2. ユーザー確認
3. 確定後にコード品質を整える

## AIの盲点（ユーザー指摘から学んだこと）

### 1. 反復のコストを認識する

ユーザーに何度も「他」「違う」と言わせるのは**失敗**。

❌ 選択肢を一つずつ出して反応を見る
✅ 最初に方向性を聞く：「シンプル/装飾的、どちらが好みですか？」

### 2. ユーザーの言葉を使う

❌ 汎用語：「件数」「項目」「データ」「要素」
✅ 具体語：「イベント数」「団体名」「日付」「カード」

**理由**: 汎用語はAIっぽい。ユーザーが使った言葉をそのまま使う。

### 3. 装飾を増やさない

「良くする」≠「装飾を増やす」

❌ タイトルに色を付ける、グラデーションを追加
✅ シンプルな細い線、控えめなアクセント

**判断基準**: 「これは本当に必要か？」と問う。

### 4. AIっぽいデザインを避ける

AIが生成しがちなパターン：
- 汎用的な青/紫グラデーション
- 予測可能なレイアウト（左右対称、均等配置）
- どこでも使えそうな金太郎飴デザイン
- Inter、Roboto等の無難なフォント

**対策**: そのコンテキストに固有の選択をする。エイサーなら沖縄らしさ、等。

### 5. レビューの限界を認識する

Codexレビューで検出**できる**：
- 型安全性、アクセシビリティ、パフォーマンス

検出**できない**：
- ユーザーの好み
- 「なんか違う」感
- ドメイン固有の適切さ

**対策**: コード品質の前に、ユーザーと方向性を合わせる。

**Codex呼び出し時のモデルルール**:
- 基本: `gpt-5.2-codex` + `reasoning: high`（必ず最初）
- エスカレーション: `reasoning: x-high`（highで不十分な場合のみ）

### 6. 聞かれていないことを提案しない

❌ 「ついでにこれも改善しましょう」
✅ 聞かれたことだけに答える

**例外**: 明らかなバグや重大な問題のみ指摘。

### 7. 凡例・ラベルは簡潔に

❌ 説明的：「イベント数を表す青いドット（1件=●、2件=●●、3件以上=●●●）」
✅ 簡潔：「●イベント数」

**原則**: ユーザーは見ればわかる。過剰な説明は不要。

### 8. モバイルリストUIの罠（フィールド系アプリで頻出）

実装後スクリーンショットで発覚しやすいパターン：

**カード名の折り返し**

長い日本語名は必ず折り返す。overflow指定を忘れると2行になり余白・FABと干渉する。

```css
/* カード内のタイトルには必ずこれ */
overflow: hidden;
text-overflow: ellipsis;
white-space: nowrap;
```

**FABの重複・重複ボタン**

タブバーに同等アクションがある場合、フローティングFABは冗長になる。両方作らない。
既に配置済みかどうかをHTML全体で確認してから追加する。

**フロー内の表記不統一**

カメラ画面→確認画面→完了画面など複数ステップがある場合、同じデータ（料金など）の表記を全ステップで揃える。

```
❌ Step1: 30分100円 → Step3: 30分/¥100
✅ 全ステップ: 30分/¥100
```

**カード内ネストCTA**

カード内にボタンをネストすると「カードタップ」と「ボタンタップ」が競合し視覚的に重い。

```
❌ カード > [エリア名] + [dashed ボタン「記録して +100pt」]
✅ カード > [エリア名] + [カメラicon 記録して +100pt] （インラインテキスト）
```

**プログレス差分の3ラベル表示**

before/diff/afterを3列で並べると意味が伝わりにくい。読む方向を揃える。

```
❌ [34%]  [+3% ↑]  [37%]   ← 何が何だかわからない
✅ 登録前 34% → 37%        +3% ↑
```

**祝福絵文字のサイズ**

成功画面の🎉などは最低 64〜72px 必要。52px以下は「小さい」と感じる。
