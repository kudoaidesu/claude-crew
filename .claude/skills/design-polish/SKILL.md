# デザインの仕上げ原則

## 核心

> 「動く」と「仕上がっている」は違う

プロの納品物には、すべての要素に**意図**がある。

## 5つの仕上げ原則

### 1. すべての要素に居場所を与える

情報をグループ化し、階層を与える。

```
タイトル（大）> 日時・場所（中）> 補足情報（小・強調色）
```

### 2. セクションに役割を与える

背景色・ボーダー・ヘッダーでゾーニングする。

```tsx
<aside className="bg-white rounded-2xl border shadow-sm">
  <div className="bg-gradient-to-r from-blue-50 to-indigo-50 border-b">
    <Icon /> ラベル
  </div>
  <Content />
  <div className="border-t bg-gray-50">{/* 凡例 */}</div>
</aside>
```

### 3. 余白を意図的に設計する

- 関連要素は近く（4-8px）
- 異なるグループは離す（16-24px）
- セクション間はさらに離す（32-48px）

### 4. インタラクションで応答性を示す

```tsx
<Link className={cn(
  "group transition-all duration-200",
  "hover:shadow-lg hover:-translate-y-0.5",
  "focus-visible:ring-2",
  "active:scale-[0.99]"
)}>
  <img className="group-hover:scale-105" />
  <h3 className="group-hover:text-blue-600" />
</Link>
```

### 5. アクセシビリティは品質の証明

- タッチターゲット: `min-h-[44px]`
- ナビゲーション状態: `aria-current="page"`
- 画像CLS防止: `width={128} height={128}`

## ⚠️ UI修正時の原則

> **UI決定を最優先、コード品質系は後回し**

UI修正中に以下を同時にやらない：
- 未使用import削除
- タイムゾーン修正
- 型修正
- リファクタリング

**理由**: UIが変わるとコード自体が不要になる可能性がある。

**正しい順序**:
1. レイアウト・構造・余白を決定
2. ユーザー確認
3. 確定後にコード品質を整える

## AIの盲点（ユーザー指摘から学んだこと）

### 1. 反復のコストを認識する

ユーザーに何度も「他」「違う」と言わせるのは**失敗**。

❌ 選択肢を一つずつ出して反応を見る
✅ 最初に方向性を聞く：「シンプル/装飾的、どちらが好みですか？」

### 2. ユーザーの言葉を使う

❌ 汎用語：「件数」「項目」「データ」「要素」
✅ 具体語：「イベント数」「団体名」「日付」「カード」

**理由**: 汎用語はAIっぽい。ユーザーが使った言葉をそのまま使う。

### 3. 装飾を増やさない

「良くする」≠「装飾を増やす」

❌ タイトルに色を付ける、グラデーションを追加
✅ シンプルな細い線、控えめなアクセント

**判断基準**: 「これは本当に必要か？」と問う。

### 4. AIっぽいデザインを避ける

AIが生成しがちなパターン：
- 汎用的な青/紫グラデーション
- 予測可能なレイアウト（左右対称、均等配置）
- どこでも使えそうな金太郎飴デザイン
- Inter、Roboto等の無難なフォント

**対策**: そのコンテキストに固有の選択をする。エイサーなら沖縄らしさ、等。

### 5. レビューの限界を認識する

Codexレビューで検出**できる**：
- 型安全性、アクセシビリティ、パフォーマンス

検出**できない**：
- ユーザーの好み
- 「なんか違う」感
- ドメイン固有の適切さ

**対策**: コード品質の前に、ユーザーと方向性を合わせる。

**Codex呼び出し時のモデルルール**:
- 基本: `gpt-5.2-codex` + `reasoning: high`（必ず最初）
- エスカレーション: `reasoning: x-high`（highで不十分な場合のみ）

### 6. 聞かれていないことを提案しない

❌ 「ついでにこれも改善しましょう」
✅ 聞かれたことだけに答える

**例外**: 明らかなバグや重大な問題のみ指摘。

### 7. 凡例・ラベルは簡潔に

❌ 説明的：「イベント数を表す青いドット（1件=●、2件=●●、3件以上=●●●）」
✅ 簡潔：「●イベント数」

**原則**: ユーザーは見ればわかる。過剰な説明は不要。
